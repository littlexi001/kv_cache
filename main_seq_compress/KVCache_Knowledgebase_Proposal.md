# Proposal: 将 KV Cache 视为“可索引知识库”的压缩与分层存储思路

Date: 2026-05-06

## 1. 问题陈述：为什么 KV Cache 问题成立

在现有 Transformer 推理中，KV cache 用于保存历史 token 在每一层（layer）每个注意力头（head）产生的 Key/Value，使得后续 token 的注意力可以复用历史状态，避免重复计算。

当上下文变长时，KV cache 的两个性质使其成为系统瓶颈：

- **空间线性增长**：序列长度每增加 1 个 token，几乎每层每头都要新增一份 K/V；长上下文推理时 GPU HBM 很快被占满。
- **带宽与延迟压力**：即便注意力算子计算被优化，读取/写入大体量 KV 也会成为吞吐与延迟的主导因素。

更关键的是：从“功能”角度看，长上下文的价值来自**检索 + 多跳组合**——模型需要在海量历史中定位少量真正相关的信息并据此推理。但现有结构在注意力访问上近似“全量扫描”，其成本与 KV 规模强绑定。

因此，KV cache 问题不是简单的“存不下”，而是：

> 我们如何在不损失关键检索能力的前提下，避免让模型在每一步都付出访问全部历史 KV 的代价？

## 2. 设计原则（Design Principle）：从人类使用知识库的方式理解 KV

### 2.1 将 KV cache 视为“知识库”（不要求是知识图谱）

我们提出的核心视角是：

> **KV cache 是一种在推理过程中持续增长的“外部记忆”**。

它不是静态的、显式的符号知识库，但它承担了类似功能：为当前 query（当前 token 在各层各头的 query 向量）提供对过去信息的可访问性。

“知识图谱”只是知识库的一种形式，这里不强制要求把 KV 变成显式实体-关系结构。我们更关注的是：把 KV 视为**可检索的记忆条目集合**，并为其提供“索引与分层存储”。

### 2.2 人类如何利用十万字大部头

人类从一本十万字的大部头里找信息，通常不会从头读到尾，而会采用“分层索引 + 快速跳转”的策略：

- **目录/索引**：先通过目录、关键词索引缩小候选范围。
- **快速扫标题与小节**：通过标题层级判断相关性。
- **局部精读**：只对少量候选段落进行细读与交叉引用。
- **分层存放**：常用知识放在随手可得处，不常用知识可放远（书架/档案），需要时再取。

这套行为模式可以抽象为：

> 先用廉价的“粗检索/路由”锁定少量候选，再用昂贵的“精检索/精算”完成推理。

这正是我们希望模型在 KV 访问上具备的能力。

## 3. 现有模型结构为什么没实现“索引式访问”

### 3.1 Dense attention 的访问模式更像“逐页扫描”

标准注意力在每一步会对历史所有 token 进行相似度计算（或等价的数据访问），其行为更接近：

- 没有目录
- 没有倒排索引
- 只能对整本书逐页扫一遍

即便有 FlashAttention 等实现优化，核心访问模式仍然是“对所有历史位置计算/读取”。

### 3.2 KV cache 的形态是“激活记忆”，而非自然语言片段

KV 是模型内部表征，具备强烈的 layer/head 语义与分布特性。

这意味着“索引式访问”不能简单等价于把文本做向量检索，而需要考虑：

- 不同 head 可能承担不同的路由/检索角色
- 同一 token 的不同层表征差异巨大
- 有效检索信号稀疏且随 query 改变

### 3.3 训练目标隐式鼓励“可用即可”，而非“可索引”

在常规训练中，只要 dense attention 能把信息传递到位，模型并不会被显式激励去学习“先定位再精算”的行为。

因此，想要实现索引式访问，往往需要：

- 结构上提供路由/索引模块
- 训练上提供对长上下文检索可靠性的明确激励（例如长上下文 RL、对证据定位的奖励等）

## 4. 如果实现“索引式访问”，为什么就能压缩/分层 KV

将 KV cache 看作知识库后，压缩不再只意味着“把每条记忆变小”，而是重新定义为：

> **让大部分记忆不必被每一步访问**。

一旦模型的访问模式变为“先选少量候选，再做精算”，我们就获得了一系列系统与算法自由度：

- **冷数据可 offload**：不被选中的大部分历史 KV 可以放在 CPU DRAM，甚至更冷的层级（disk），只在命中时调回。
- **KV 可按块管理**：以 block/page 为粒度维护热度与驻留，提高搬运与预取效率。
- **压缩可更激进**：对冷数据可采用更强量化/合并，因为它们对最终输出的边际贡献更低。
- **成本与有效信息解耦**：计算/带宽成本主要由“被选中的候选数”决定，而不是由“总长度”决定。

这与“长上下文并非更大窗口，而是更大的可用窗口”的目标一致。

## 5. 技术方案（Solution）

下面按“改动幅度从小到大”列出若干方向，便于后续讨论与取舍。

### 5.1 块级两阶段注意力：粗路由选块 + 块内精算

- 将历史 token 按固定大小划分为 blocks/pages
- 用廉价的 block 表征（例如 block summary K 或 centroid）先做 top-k blocks 选择
- 仅对命中的 blocks 展开 token-level 注意力

优点：容易与分页 KV 管理结合，可直接带来“冷块不访问”的系统收益。

### 5.2 近似最近邻（ANN）索引：为 K 建立可更新索引

- 为历史 K 构建可动态插入的 ANN 结构（HNSW/IVF/LSH 等）
- query 先在索引上检索候选 key，再做精确注意力

挑战：动态更新、GPU/CPU 混合索引、以及多头多层的并行与一致性。

### 5.3 分层记忆：HBM/DRAM/disk 的 KV 分层与预取

- 以 block 为单位维护热度（recency/frequency/utility）
- HBM 保热块，DRAM 保温块，disk 保冷块
- 命中路径要求异步搬运 + 预取 + 计算/拷贝重叠

注：disk 层通常只适合极冷、可容忍高尾延迟的场景；更现实的第一步是 HBM↔DRAM。

### 5.4 冷数据更强压缩：量化、低秩、合并成“记忆槽”

- 对温/冷 KV 做更激进量化（例如更低比特）
- 将多个 token 的 KV 合并为少量 memory slots（保代表性信息）
- 允许必要时局部重算（若具备重放输入或可重构中间表征）

关键点：压缩与选择（routing）应联动，冷数据被访问概率低时可更激进。

### 5.5 训练驱动的可索引行为：让模型学会“先找证据”

- 通过监督/强化学习让模型在长上下文下显式依赖远处证据
- 增加对“证据定位正确性/覆盖率”的奖励信号
- 让部分头/层承担路由职责（检索头），形成结构化分工

## 6. 评估框架（用于验证洞见是否成立）

为了避免过早陷入实现细节，建议用少量关键指标验证“索引式访问”的核心收益：

- **质量**：长上下文检索/多跳任务（needle、multi-hop、代码定位等）在稀疏访问下的退化曲线
- **稀疏度**：每步实际访问的 token/block 比例、top-k 命中分布
- **系统性能**：prefill/decoding 延迟、吞吐、HBM 峰值、PCIe/NVLink 带宽占用
- **尾延迟**：offload/回载路径对 P95/P99 的影响

## 7. 小结

本 proposal 的核心不在于具体实现，而在于把问题从“把 KV 变小”升级为“改变访问模式”。

将 KV cache 视为可索引知识库后，我们的目标是让模型像人类查书一样：先用廉价索引定位候选，再对少量内容精读，从而为 KV 的压缩、分层存储与按需回载创造条件。
